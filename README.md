# Multi-Service Security System (Cyber Lab 2)

Проект представляет собой комплексную систему информационной безопасности, реализующую управление секретами (Vault) и сервис цифровой подписи. Система построена на микросервисной архитектуре с использованием современных методов криптографии.


## Архитектура проекта

Проект разделен на три ключевых компонента:

1.  **Backend Vault (Порт 8003)**: Управляет жизненным циклом мастер-ключа и хранением зашифрованных секретов.
2.  **Backend Signature (Порт 8001)**: Отвечает за вычисление хеш-сумм и формирование цифровых подписей файлов.
3.  **Frontend**: Веб-интерфейс для взаимодействия с сервисами без использования сторонних инструментов типа Postman.

## Основные функции

### 1. Схема разделения секрета (Shamir-like/XOR)
Для обеспечения высокой доступности и безопасности мастер-ключ не хранится в открытом виде. Он восстанавливается в оперативной памяти только после ввода **двух независимых частей ключа**.
*   **Статус "Sealed"**: Хранилище заблокировано, доступ к секретам невозможен.
*   **Статус "Unsealing"**: Получена первая часть ключа.
*   **Статус "Unsealed"**: Мастер-ключ восстановлен, функции шифрования активны.
*   **Процесс**:
    * `POST /init` генерирует случайный мастер-ключ (32 байта), делит его на 2 части через XOR и **возвращает обе части клиенту** (сервер не хранит мастер-ключ в явном виде).
    * `POST /unseal` принимает части по одной; когда введены обе, мастер-ключ восстанавливается **в памяти** и Vault становится `unsealed`.
    * `POST /seal` удаляет мастер-ключ из памяти и переводит Vault обратно в `sealed`.

### 2. Хранилище секретов
*   **Алгоритм**: Fernet (симметричное шифрование на базе AES-128 в режиме CBC с HMAC SHA256).
*   **Хранение**: Файл `vault_storage.json`, данные в котором защищены мастер-ключом.
*   **Wrap/Unwrap с TTL**:
    * `POST /wrap` шифрует секрет мастер-ключом и выдаёт одноразовый токен на 60 секунд.
    * `POST /unwrap` по токену возвращает расшифрованный секрет (и удаляет токен).
    * **Что лежит в токене**: токен — это идентификатор в памяти сервиса Vault. По токену временно хранится **ciphertext** (зашифрованный Fernet-ом секрет) и время истечения TTL. При `unwrap` запись удаляется (одноразовое извлечение).

### 3. Цифровая подпись
*   **Реализация (6 баллов)**: документ **не изменяется**. К нему добавляются отдельные **метаданные подписи** (JSON), которые содержат подпись, публичный ключ и параметры алгоритма.
*   **Алгоритм подписи**: RSA-PSS или Ed25519 (выбор на клиенте).
*   **Хеширование**:
    * для RSA-PSS: SHA-256 или SHA-512 (выбор на клиенте)
    * для Ed25519: хеширование встроено в алгоритм
*   **API**:
    * `POST /sign` принимает файл и возвращает JSON-метаданные подписи.
    * `POST /verify` принимает файл + JSON-метаданные и возвращает `valid: true/false`.
*   **Бонус (демо слабого хеша)** — учебный хеш `weak_hash = sum(bytes) mod 256`, **не используется** в реальной подписи RSA-PSS / Ed25519:
    * `POST /demo/weak-hash` — загрузить **свой файл** (`multipart/form-data`, поле `file`); ответ содержит `weak_hash_sum256` и размер файла.
    * `POST /demo/weak-hash/forge` — тот же файл; сервер возвращает JSON с хешами для оригинала и подделки (хеш совпадает), поле `forged_file_b64` (base64 изменённого файла: к содержимому дописаны байты `01 FF`, сумма которых даёт `0 mod 256`). Так показывается, что при таком «хеше» файл можно изменить без смены значения хеша.

Тот же сценарий доступен во фронтенде: блок «Демо уязвимости слабого хеша» вызывает эти два эндпоинта.

## Установка и запуск

### Предварительные требования
*   Python 3.9+
*   Библиотеки: `fastapi`, `uvicorn`, `cryptography`, `pydantic`

### Инструкция по запуску

1. **Запустите сервис Vault**:
   ```bash
   cd backend_vault
   python main.py
   ```

2. **Запустите сервис подписи**:
   ```bash
   cd backend_signature
   python main.py
   ```

3. **Откройте Frontend**:
   Откройте файл `frontend/index.html` в браузере.
